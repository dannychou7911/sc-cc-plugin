# L4 — 程式碼內文件模板與規範

> **目標讀者**：直接閱讀程式碼的工程師
> **更新頻率**：隨程式碼修改即時更新
> **載體**：程式碼中的註解、JSDoc/docstring、型別

L4 不產出獨立的文件檔案，而是以程式碼註解和型別標註的形式存在於 codebase 中。

---

## 核心原則

> 程式碼註解記錄 **Why**（為什麼這樣做）和 **What for**（為了什麼目的），
> 不記錄 **What**（做了什麼）— What 由程式碼本身表達。

**好的註解**：

```
// 送審後不可刪除，2024-06 客戶要求（JIRA: OMS-342）
if (order.status !== 'draft') return false;
```

**壞的註解**：

```
// 檢查訂單狀態是否為 draft
if (order.status !== 'draft') return false;
```

## 標記慣例

| 標記     | 用途                              | 格式                                    |
| -------- | --------------------------------- | --------------------------------------- |
| `TODO:`  | 已知需改善但非緊急                | `// TODO: {描述} ({日期或 ticket})`     |
| `FIXME:` | 已知 bug 或有問題的實作           | `// FIXME: {描述} ({日期或 ticket})`    |
| `HACK:`  | 刻意的 workaround，需記錄原因    | `// HACK: {原因}。移除條件：{何時可移除}` |
| `NOTE:`  | 重要但不明顯的邏輯說明            | `// NOTE: {說明}`                       |
| `@see`   | 指向相關文件或 issue              | `// @see {文件路徑或 URL}`              |
| `@since` | 標記功能新增的時間點              | `// @since {YYYY-MM} {ticket}`          |

## 應加上註解的程式碼模式

在分析程式碼時，遇到以下模式應建議加上註解：

### 1. 業務規則的條件判斷

任何不看業務背景就無法理解的 if 判斷。

### 2. 非直覺的資料轉換

Request / response 的資料格式轉換，特別是前後端格式不一致的地方。

### 3. 效能相關的決策

選擇特定資料結構或 API 的效能考量。

### 4. Workaround

暫時性的解法，必須標記原因、移除條件、相關 issue。

### 5. 刻意不做的事（Negative Design Decision）

有意識地選擇不做某件事（如不使用 optimistic update），記錄原因。

---

## 型別作為活文件

型別定義本身就是規格文件。讓型別發揮最大文件價值的原則：

- 使用 Union Type / Enum 取代裸字串（型別本身就是規格）
- 使用具名常數取代 Magic Number
- 為複雜型別的欄位加上說明（特別是格式、來源、生命週期）

## 活文件維護策略

Phase 5 的最終產出應包含一個「活文件維護建議」區塊，根據偵測到的技術棧推薦具體做法。

### 通用策略

- 規格文件放在 `docs/` 目錄中版控，與程式碼一起 review
- PR template 加入 checkbox：「此 PR 是否需要更新規格文件？」
- 重大行為變更的 commit message 中標記 `[spec-update-needed]`

### 前端專案

- TypeScript 嚴格模式 + Union Type 作為狀態/列舉的活規格
- Storybook（如有使用）作為 UI 元件的活規格
- E2E 測試描述作為使用者行為的活規格

### 後端專案

- OpenAPI / Swagger 自動產生（如 NestJS @ApiProperty）作為 API 活規格
- DB migration 檔案作為 schema 演進紀錄
- 整合測試描述作為業務邏輯的活規格

### 文件健康度指標

建議在規格文件中加入「最後驗證日期」欄位，定期檢查：

| 指標 | 健康 | 警告 | 危險 |
|------|------|------|------|
| 最後驗證距今 | < 3 個月 | 3–6 個月 | > 6 個月 |
| 對應程式碼修改次數 | 0 次 | 1–5 次 | > 5 次 |

## 測試接縫（Test Seam）建議

在分析過程中標記無測試覆蓋的高風險區域，產出建議清單：

```markdown
## 測試接縫建議

| 優先級 | 模組 | 風險描述 | 建議測試類型 | 接縫位置 |
|--------|------|---------|-------------|---------|
| 高     | {模組} | {為什麼高風險} | {unit / integration / e2e} | {可插入測試的位置} |
```

高風險判定標準：
- 涉及數值計算
- 涉及權限控制
- 狀態機轉換邏輯
- 複雜的條件分支（> 3 個分支）
- 跨模組的資料依賴

## 產出格式

L4 的產出以「建議清單」形式呈現：

```markdown
# L4 程式碼文件建議

## {檔案路徑}

### 建議 1：{行號範圍}

**類型**：{JSDoc / 行內註解 / 型別改善 / 標記}
**優先級**：{高 / 中 / 低}
**理由**：{為什麼需要加這個註解}

建議新增：
```
// {建議的註解內容}
```
```

或直接以 diff 格式呈現，方便直接套用。
