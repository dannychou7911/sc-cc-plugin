# 結構化 Q&A 協議

每個 Phase 結束後，依序執行：
1. **待確認項目確認**（第一部分）
2. **業務脈絡收集**（第二部分）
3. **補充 Q&A**（第三部分）
4. **Context 壓縮檢查**（第四部分）

---

## 原則

**AI 先寫，人來修正。** 永遠不問程式碼已經能回答的問題。每個問題都必須是程式碼真正無法揭示的——業務意圖、命名理由、已棄用功能、未來計畫、模糊邏輯。

---

## 工具要求

**所有需要使用者回答的問題，必須使用 `AskUserQuestion` 工具提問。** 不要用普通文字輸出提問。

原因：
- `AskUserQuestion` 會在使用者介面產生明確的輸入區塊，方便使用者辨識並回應
- 避免問題被淹沒在大量分析文字中
- 使用者可以清楚看到哪些地方需要他的輸入

使用方式：
- 先用普通文字輸出分析摘要和上下文
- 然後用 `AskUserQuestion` 工具提出需要使用者回答的問題
- 將所有問題（3-5 個）組合成一次 `AskUserQuestion` 呼叫，不要分多次問

---

## 第一部分：待確認項目確認

> Phase 結束流程的第一步。

Phase 產出文件後，掃描文件中所有 `[待確認]` 標記項目，使用 `AskUserQuestion` 逐一向使用者確認。

**執行步驟**：
1. 列出文件中所有 `[待確認]` 項目清單
2. 每輪用 `AskUserQuestion` 提出 3-4 個待確認項目（附選項）
3. 根據使用者回答，更新文件中的標記：
   - 使用者給出明確答案 → 更新為 `[確認]`，補充內容
   - 使用者回答「不確定」→ 更新為 `[未確定]`，移入 Open Questions
4. 重複直到所有 `[待確認]` 項目都被處理

**範例**：
```
文件中發現：
> helmet.frameguard: false 的目的 [待確認]

AskUserQuestion：
「helmet.frameguard: false 關閉了 iframe 嵌入防護，這是刻意的設計嗎？」
選項：
(A) 是，業務需要 iframe 嵌入（請說明場景）
(B) 不確定，可能是歷史遺留
(C) 其他
```

## 第二部分：業務脈絡收集

程式碼能揭示 what，但部分 why 完全不存在於程式碼中。主動識別這類盲區，用 `AskUserQuestion` 向使用者收集業務脈絡。

### 觸發時機

| 時機 | 收集範圍 | 說明 |
|------|---------|------|
| Phase 1 Q&A 時 | 全局業務脈絡 | 系統地圖完成後，已知模組清單和技術棧，可以問全局性問題 |
| Phase 3 每個模組開始前 | 模組業務語義 | 進入功能拆解前，先理解該模組的領域術語和業務規則 |

### 應收集的類別

根據分析結果，**動態產生**以下類別的問題（非固定問題清單）：

| 類別 | 觸發條件（程式碼中觀察到的線索） |
|------|-------------------------------|
| **業務領域與法規** | 發現看似受外部規範約束的設計模式（如特定流程、格式、限制），但程式碼中無註解說明來源 |
| **領域術語語義** | 發現業務實體名稱（如角色、狀態、類型），但程式碼中只有英文變數名或 magic value |
| **已廢棄但保留的功能** | 發現有入口但無路由指向、有程式碼但無 UI 觸發、或有註解暗示棄用 |
| **外部系統約束** | 發現特定格式轉換、固定欄位名稱、非標準協議，可能源自第三方系統要求 |
| **跨團隊隱性契約** | 發現 API 有多個消費者跡象（如版本化、相容性處理、多格式回傳） |
| **商業策略性限制** | 發現硬編碼的數量限制、層級限制、功能開關，但無程式碼註解說明原因 |

### 原則

- **只問程式碼無法回答的問題**：能從程式碼確認的行為不要問
- **問題必須基於分析發現**：先分析程式碼，再根據發現的線索產生問題，而非使用固定問題清單
- **「不知道」是合法答案**：標記為 `[未確定-業務脈絡]`，與一般 `[未確定]` 區分，代表業務知識缺失而非程式碼分析不足
- **記錄到工作偏好**：使用者提供的業務脈絡記錄到進度檔「工作偏好」區塊，後續模組分析時參考

## 第三部分：補充 Q&A 流程

### 1. 先展示摘要

在提問之前，先展示當前 Phase 的分析摘要，讓使用者有上下文：

```
## Phase {N} 分析摘要
- **已確認項目**：X 個
- **AI 推斷項目**：Y 個
- **需確認項目**：Z 個
```

### 2. 一次一個模組

展示該模組的草稿摘要作為問題的上下文。不要跨模組混問。

### 3. 每輪 3-5 個問題

永遠不超過 5 個。問題太多會讓使用者疲勞。

### 4. 提供選項，附優缺點評估

每個選項必須在 `description` 中包含**優缺點或影響分析**，幫助使用者快速判斷。使用者通常不熟悉所有技術細節，提供評估讓他們做出有依據的選擇。

**格式**：
```
AskUserQuestion option:
  label: "選項名稱"
  description: "優點：{好處}。缺點/風險：{壞處}。適用於：{場景}"
```

**範例**：
```
/admin 路由的用途：
(A) 後台管理介面（僅管理員）
    → 優點：符合最小權限原則。缺點：需要額外的角色判斷邏輯
(B) 系統設定頁面（所有登入使用者）
    → 優點：簡化權限管理。缺點：敏感設定可能被非管理員存取
(C) 其他：___
```

### 5. 優先問高影響問題

優先順序：
1. 業務意圖（程式碼無法揭示的 why）
2. 模糊命名或邏輯
3. 死碼 / 已棄用功能
4. 權限規則和邊界案例
5. 外部整合目的
6. 設計決策的理由（ADR 補充）

### 6. 跳過瑣碎確認

如果程式碼顯示 `required` + `maxlength=50`，不需要問。

### 6.1 可跳過結構化 Q&A 的條件

以下情境中，Phase 結束後可跳過補充 Q&A，僅執行待確認項目確認：

- **分析範圍極小**：單一檔案或單一函式的規格回推（非整個模組）
- **無待確認項目**：Phase 產出文件中無任何 `[待確認]` 或 `[推斷]` 標記
- **連續 Phase 無新發現**：前一輪 Q&A 已覆蓋所有疑問，本 Phase 未產生新的業務意圖問題
- **使用者明確要求跳過**：使用者表示「不需要問了」或「直接繼續」

跳過時仍須在進度檔標記 `Q&A：已跳過（原因：{原因}）`。

### 7. 交叉比對答案與程式碼

如果使用者的回答與程式碼矛盾，主動標記：

```
您提到此模組不需登入，但 auth guard 涵蓋此路由。
這是歷史遺留需清除，還是確實需要登入？
```

### 8. 追蹤進度

每輪結束後顯示：

```
已確認: 使用者管理、商品列表
進行中: 訂單管理 (3/5)
待確認: 報表、系統設定
```

### 9. 「我不知道」是合法答案

標記為 `[未確定]`，加入 Open Questions appendix，繼續下一題。

### 10. 大型專案策略

超過 15 個模組時：
1. 請使用者排優先級
2. 從最高優先級模組開始
3. 提供後續 session 處理其餘模組的選項

---

## 第四部分：Context 壓縮檢查

待確認項目確認和 Q&A 完成後，執行壓縮檢查：

1. 使用 `strategic-compact` skill 評估目前 context 使用量
2. 若建議壓縮，執行 `/compact` 命令
3. **前置條件檢查**：若使用者環境缺少以下任一項，提醒補上後再繼續：
   - `strategic-compact` skill（用於評估壓縮時機）
   - `/compact` command（用於執行壓縮）
