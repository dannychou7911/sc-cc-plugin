## 方法論依據

評分維度與評分方法的設計來源：

| 方法論 | 核心概念 | 對應的評分維度/方法 |
|--------|---------|-------------------|
| IEEE 29148 | 需求品質屬性：correct, unambiguous, complete, consistent, verifiable, traceable | A 維度：完整性、準確性 |
| Specification by Example | 好的規格 = 可直接轉為測試案例 | B 可測試性評分 |
| BDD (Given-When-Then) | 行為導向、以使用者角度描述 | A 維度：完整性（行為覆蓋率）；B 行為盤點方式 |
| Architecture Decision Records | 決策記錄 why + trade-off | A 維度：Why 記錄 |
| Working Effectively with Legacy Code | 識別接縫、安全網思維 | A 維度：隱性規則覆蓋（風險識別） |
| Diátaxis | 不同目的的文件有不同品質標準 | 各層次權重矩陣的差異化設計 |
| C4 Model | 不同粒度給不同受眾 | A 維度：受眾適配 |

## 執行方式

評分由 `spec-scorer` 自訂 agent 執行（定義於 `~/.claude/agents/spec-scorer.md`），與產出文件的主 agent 分離，避免 confirmation bias。

### 執行流程

```
主 Agent（產出文件）
  ↓ Write 完成
主 Agent 讀取對應模板（templates-L{N}.md），摘要格式要求
  ↓
主 Agent 啟動 spec-scorer agent（傳入：待評文件、層次、模板摘要、原始碼目錄、是否啟動 C）
  ↓
spec-scorer agent（依內嵌評分標準執行評分）
  ↓ 回傳結構化評分結果
主 Agent（記錄到進度檔、判斷是否觸發補強）
```

> **注意**：評分標準已內嵌於 `spec-scorer` agent 定義中，agent 不需讀取本檔案。本檔案保留作為評分方法論的參考文件。

### 啟動 spec-scorer 的 Prompt 範例

主 agent 使用 Agent tool 啟動 `spec-scorer` agent 時，Prompt 需提供以下資訊。模板格式要求可引用 `template-summaries.md`，或自行讀取對應 `templates-L{N}.md` 後摘要（因 sub agent 無法讀取 skills 目錄）。

```
審閱並評分規格文件。

## 輸入
- **待評文件**：docs/L2-features/member-list.md
- **文件層次**：L2
- **原始碼目錄**：src/views/member/list/
- **啟動差異驗證**：是

## 模板格式要求
（以下為主 agent 讀取 templates-L2.md 後的摘要）
- 必須包含：Header、TOC、進入條件、初始載入、使用者操作、條件渲染、錯誤處理、隱性規則
- 流程使用 mermaid flowchart
- 隱性規則使用 🔴/🟡 分級並附 Why
- ...
```

### 差異驗證自動啟用規則

以下情境必須啟動差異驗證（C 評分），無需使用者確認：
- 涉及認證、權限、金額的高風險模組
- L2 文件（投資報酬率最高的層次，遺漏代價最大）
- A 評分 < 70 的文件

### 主 Agent 後續處理

收到評分結果後，主 agent 負責：

1. **記錄到進度檔**（`docs/spec-recovery-progress.md`）
2. **判斷行動**：
   - ≥ 70：通過，繼續下一份文件
   - 60-69：主 agent 自行補強最弱的 1-2 個維度，再啟動 `spec-scorer` agent 重新評分
   - < 60：主 agent 重做該文件
3. **人工介入判斷**：若符合「人工介入觸發條件」，使用 `AskUserQuestion` 請使用者介入

### 修正流程細節

1. 從評分結果中識別**具體弱項**（哪個維度扣分、扣分原因）
2. 回到原始碼重新驗證弱項相關的行為（不憑記憶修正）
3. 更新規格文件中對應的段落
4. 啟動新的 `spec-scorer` agent 重新評分（不可自我評分）
5. 將**最終分數**記錄到 `spec-recovery-progress.md` 的對應表格（格式：`A:XX%` 或 `A:XX% B:XX%`）
6. 若二次評分仍 < 70，使用 `AskUserQuestion` 請人工介入

## A. 通用多維度評分

每份文件在 6 個維度各打 1-5 分，依層次加權計算總分。

### 維度定義

| 維度 | 1 分 | 3 分 | 5 分 |
|------|------|------|------|
| **完整性** | 遺漏多項行為或功能 | 主要行為已覆蓋，少量邊界情況缺漏 | 所有行為、邊界情況、錯誤情境均已記錄 |
| **準確性** | 描述與程式碼行為明顯不符 | 大致正確，少數細節有誤差 | 每項描述均可對應到具體程式碼，無誤差 |
| **Why 記錄** | 僅記錄 what，無任何 why | 部分設計決策有 why，但多數僅描述現象 | 關鍵決策均記錄 why + trade-off，含 Negative Design Decision |
| **隱性規則覆蓋** | 無標記任何隱性規則 | 找到部分 magic value 和硬編碼，但遺漏重要契約 | 所有 magic value、前後端契約、順序依賴均標記並正確分級 🔴/🟡 |
| **格式規範** | 多處不符模板（缺 TOC、無 mermaid、缺 Header） | 基本結構符合，少數流程未用 mermaid | 完全符合模板、TOC 完整、所有流程用 mermaid、Confidence Marker 標記完整 |
| **受眾適配** | 目標讀者無法理解或無法直接使用 | 多數內容對目標讀者有用，但部分需補充才能行動 | 目標讀者可直接依據文件行動（QA 寫測試、新人理解系統、PM 掌握全貌） |

### 各層次權重矩陣

| 維度 | L0 | L1 | L2 | L3 | L4 |
|------|-----|-----|-----|-----|-----|
| 完整性 | 25% | 25% | 20% | 25% | 20% |
| 準確性 | 10% | 20% | 20% | 25% | 20% |
| Why 記錄 | 25% | 15% | 10% | 10% | 30% |
| 隱性規則覆蓋 | 10% | 15% | 25% | 20% | 15% |
| 格式規範 | 15% | 10% | 10% | 10% | 5% |
| 受眾適配 | 15% | 15% | 15% | 10% | 10% |

**權重設計理由**：
- **L0**：給所有人看的鳥瞰圖，Why（為何這樣架構）和受眾適配最重要
- **L1**：模組邊界需精確，完整性和準確性並重
- **L2**：ROI 最高的層次，隱性規則覆蓋是核心價值（QA 最怕踩到未記錄的地雷）
- **L3**：API 契約需精確，準確性權重最高
- **L4**：程式碼註解的核心是 Why，佔 30%

### 計算方式

```
加權總分 = Σ(維度分數 × 維度權重)
百分制 = 加權總分 / 5 × 100
```

## B. 可測試性評分（僅 L2）

基於 Specification by Example：一份好的 L2 文件，QA 不需讀 code 就能寫測試案例。

### 評估步驟

1. **盤點行為數**：統計文件中描述的所有離散行為（一個操作 + 一個預期結果 = 一個行為）
2. **衍生測試案例數**：判斷有多少行為可直接轉為 Given-When-Then 測試案例（含成功路徑 + 錯誤路徑 + 邊界情況）
3. **計算覆蓋率**

```
可測試性 = 可衍生測試案例數 / 行為描述數 × 100%
```

### 可測試性判定標準

| 等級 | 覆蓋率 | 說明 |
|------|--------|------|
| 優秀 | ≥ 90% | 幾乎所有行為都有明確的觸發條件和預期結果 |
| 良好 | 70-89% | 主要行為可測試，少數描述過於模糊 |
| 需改善 | 50-69% | 多數行為缺少觸發條件或預期結果 |
| 不合格 | < 50% | 文件描述以解釋為主，無法轉為測試 |

### 不可測試的常見原因

- 描述僅有「做了什麼」，缺少「在什麼條件下」和「預期結果是什麼」
- 使用模糊詞（「適當地」「正確地」）而非具體條件
- 缺少錯誤路徑和邊界情況
- 條件渲染規則未列出完整的條件組合

## C. 差異驗證評分（可選）

重新讀取程式碼，比對「規格描述 vs 程式碼實際行為」，找出遺漏。

### 適用時機

- 高風險模組（涉及數值、權限、認證等敏感資料）
- A 評分 < 70 的文件（作為診斷工具）
- 使用者明確要求深度驗證

### 執行步驟

1. **程式碼行為盤點**：重新讀取模組程式碼，列出所有離散行為（不參考已產出的規格文件）
2. **規格覆蓋比對**：逐一檢查每個程式碼行為是否被規格文件記錄
3. **計算覆蓋率並列出遺漏**

```
規格覆蓋率 = 規格已記錄行為數 / 程式碼實際行為數 × 100%
```

### 比對輸出

```markdown
### 差異驗證結果

| 指標 | 數值 |
|------|------|
| 程式碼行為數 | {N} |
| 規格已覆蓋 | {M} |
| 覆蓋率 | {M/N × 100}% |

#### 遺漏項

| # | 程式碼行為 | 位置 | 風險等級 | 建議 |
|---|-----------|------|---------|------|
| 1 | {遺漏的行為描述} | {檔案:行號} | 高/中/低 | 補入 L2 / 可忽略 |
```

### 覆蓋率判定標準

| 等級 | 覆蓋率 | 行動 |
|------|--------|------|
| 優秀 | ≥ 90% | 無需修正 |
| 良好 | 75-89% | 補充遺漏的高風險項 |
| 需改善 | < 75% | 回頭重新分析遺漏區域 |

## 評分等級與行動

### 綜合判定（A 為主，B/C 為輔）

| 總分 | 等級 | 行動 |
|------|------|------|
| ≥ 80 | 優秀 | 通過，記錄分數進入進度檔 |
| 70-79 | 良好 | 通過，標記弱項維度供未來改善 |
| 60-69 | 需改善 | **觸發補強**：針對最低分的 1-2 個維度回頭補充 |
| < 60 | 不合格 | **觸發重做**：重新分析該文件，對照弱項維度逐一改善 |

### 補強流程

當 A 總分 < 70 時：

1. 識別最低分的 2 個維度
2. 重新讀取相關程式碼，針對弱項補充
3. 更新文件後重新評分
4. 若仍 < 70，啟動 C（差異驗證）找出具體遺漏

### 人工介入觸發條件

以下情況建議使用 `AskUserQuestion` 請人工介入：

- A 評分中「準確性」< 3 且有 `[推斷]` 標記的項目（AI 不確定是否正確）
- B 可測試性 < 50%（文件可能方向錯誤，非細節問題）
- C 差異驗證發現 ≥ 3 個高風險遺漏（可能影響架構理解）
- 兩次補強後總分仍 < 70（可能需要業務知識才能改善）

## 進度檔記錄格式

評分結果記錄到 `spec-recovery-progress.md` 的對應項目中：

```markdown
| 功能/頁面 | 狀態 | 產出檔案 | 評分 |
|----------|------|---------|------|
| 登入 | ✅ 完成 | `L2-features/auth-login.md` | A:82% B:85% |
```

跨模組比較時，在進度檔底部新增彙總區塊：

```markdown
## 評分彙總

| 模組 | L0 | L1 | L2 (avg) | L3 | L4 | 可測試性 (avg) |
|------|----|----|----------|----|----|---------------|
| Auth | 78% | 82% | 85% | 80% | 76% | 87% |
| Member | - | 80% | 78% | - | - | 75% |
```
