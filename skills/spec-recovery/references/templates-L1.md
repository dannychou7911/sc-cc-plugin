## 產出指引

### 模組邊界判定方法

模組邊界常常不清楚，以下是判定的經驗法則：

1. **路由分群**：同一路由前綴下的頁面通常屬於同一模組
2. **Store 歸屬**：一個 store 通常只服務一個模組
3. **獨立可部署性**（思想實驗）：如果把這個模組整個拿掉，其他模組能否正常運作？如果能，邊界就切對了
4. **領域語言一致性**：模組內的命名應一致。如果混用了不同領域的術語，可能需要拆分

### 資料流分析步驟

1. 從頁面/入口的初始化邏輯開始，追蹤初始載入做了什麼
2. 追蹤每個 action 的呼叫鏈：action → service → HTTP client
3. 追蹤 state 被哪些 computed / derived state 消費
4. 用箭頭串起來就是資料流

### 狀態機萃取

搜尋以下模式來識別隱含的狀態機：

- `status` / `state` 欄位的 enum 或 union type 定義
- 對 `status` 做條件判斷的所有 `if` / `switch` 語句
- 根據 status 決定顯示哪些操作按鈕的邏輯
- API 中用於變更狀態的端點（如 `POST /orders/:id/submit`）

將收集到的資訊繪製為 `狀態 A --[操作/事件]--> 狀態 B`，確認每個狀態都有至少一個進入路徑，並標記終態。

### 依賴關係識別

掃描模組內所有檔案的 import 語句，如果 import 指向另一個模組的目錄，記錄為「對外依賴」。

### 設計決策萃取

以下程式碼模式通常暗示著設計決策：

- `// HACK:` / `// FIXME:` / `// TODO:` 註解 → 已知的妥協
- 異常複雜的邏輯（嵌套 if > 3 層）→ 可能是 workaround
- 被註解掉的程式碼 → 曾經的嘗試或棄用功能
- 自行實作常見功能（如手寫 debounce 而非用 lodash）→ 有特殊原因？
- 跨模組的直接 import（而非透過 public API）→ 架構邊界破壞
