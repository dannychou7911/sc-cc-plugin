## 操作規則

### 何時建立

Phase 1 完成後建立進度檔，包含：
- Phase 0、1 的完成狀態
- Phase 2 的模組清單（從 L0 文件中提取）
- Phase 3 的頁面清單留空，待 Phase 2 完成後補充

### 何時更新

**每次產出檔案後，必須立即更新進度檔。** 這是強制規則，不可延遲或批量更新。

具體時機：
1. **每產出一份文件後**（Write 完成 → 立即更新進度檔中對應項目狀態）
2. **評分完成後**，將最終分數記錄到對應表格的「評分」欄（格式：`A:XX%` 或 `A:XX% B:XX%`）
3. **修正後重新評分**，更新為最終分數（僅保留最終分數，不保留中間分數）
4. **使用者給出工作偏好時**，記錄到「工作偏好」區塊（無日期前綴，保持精煉）
5. **發現待處理項目時**，記錄到「待處理項目」區塊（完成後移除）
6. **里程碑完成時**（Phase 完成、格式更新等），記錄到「歷史紀錄」區塊（時間倒序）
7. **模組內 Q&A 完成後**，標記該模組的 Q&A 完成狀態
8. **檢查點通過後**，將確認結果記錄到「工作偏好」區塊
7. **整個模組走完所有 Phase 後**，更新底部「評分彙總」區塊

### 恢復工作時

1. 讀取 `docs/spec-recovery-progress.md`
2. **必讀「工作偏好」和「待處理項目」**，「歷史紀錄」可跳過
3. 讀取最近完成的產出檔案（作為格式參考）
4. 從第一個 ⬚ 或 🔄 項目繼續

### Phase 3-5 任務清單補充時機

Phase 2 完成後，根據各 L1 文件中的頁面清單，展開 Phase 3-5 的任務清單。

**逐模組 Q&A 追蹤**：工作流是逐模組走完 Phase 3→4→5（如 Auth 先走完，再走 Member），因此 Q&A 追蹤放在**模組層級**（如 `### Auth 模組` 下方），而非 Phase 層級。

格式：

```markdown
## Phase 3: 功能行為拆解 (L2)

### Auth 模組

| 功能/頁面 | 狀態 | 產出檔案 | 評分 |
|----------|------|---------|------|
| 登入 | ⬚ | `L2-features/auth-login.md` | |
| 個人資料 | ⬚ | `L2-features/auth-profile.md` | |

- [ ] Auth 待確認 Q&A
- [ ] Auth 結構化 Q&A

### Member 模組

| 功能/頁面 | 狀態 | 產出檔案 | 評分 |
|----------|------|---------|------|
| 會員列表 | ⬚ | `L2-features/member-list.md` | |
| 會員詳情 | ⬚ | `L2-features/member-detail.md` | |

- [ ] Member 待確認 Q&A
- [ ] Member 結構化 Q&A
```

Phase 4、5 同理，每個模組下方附帶該模組的 Q&A checkbox。

## 檢查點機制

### 首份文件檢查點

每個 Phase 的**第一份文件**完成後，暫停並使用 `AskUserQuestion` 確認：

1. **粒度**：這個詳細程度是否符合你的需求？（太詳細 / 剛好 / 太粗略）
2. **格式**：文件結構和呈現方式是否好用？
3. **方向**：內容重點是否正確？

將確認結果記錄到進度檔的「工作偏好」區塊，後續文件遵守。

### Phase 完成驗收

每個 Phase 的所有任務完成後，輸出簡短驗收摘要：

```
## Phase {N} 驗收摘要
- **產出**：{N} 份文件
- **覆蓋**：{列出已覆蓋的模組/頁面}
- **未覆蓋**：{列出未處理的項目及原因}
- **待確認項目**：{N} 個
```

使用 `AskUserQuestion` 確認是否可以進入下一個 Phase。
